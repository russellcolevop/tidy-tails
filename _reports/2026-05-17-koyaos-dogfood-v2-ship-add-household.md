# KoyaOS dogfood lesson — Tidy Tails v2 Add Household (third write surface)

Earlier sessions produced REQ-01 through REQ-29. Building Add Household — the
venture's third dry-run/gated write surface, and the first that writes more
than a single row — surfaced one more.

---

## REQ-30 — The dry-run model can't fully prove a dependent multi-row write; classify and gate those surfaces separately

**What happened:** Add Household is Tidy Tails v2's third dry-run/gated write
surface, built — like M2 (REQ-28) and Log Groom (REQ-29) — so the complete flow
runs (auth, validation, payload construction) and nothing persists. But it is
the first surface that writes **more than one row, across two tables, with a
dependency**: onboarding a household is a `clients` INSERT followed by a `pets`
INSERT whose `client_id` is the id the first insert *generates*. The dry-run
builds both payloads — and `buildPetInsert` deliberately omits `client_id`,
because that value is structurally unknowable until the client row exists.

**Pattern that worked:** one payload builder per table (`buildClientInsert`,
`buildPetInsert`), with the child builder omitting the foreign key the parent
insert generates — explicitly and with a comment, never faked with a placeholder
id. The dry-run proves every column it *can*; the one it can't is visible.

**Why it matters:** REQ-28's promise was "build the payload so the shape is
proven before the gate lifts." That holds cleanly for a single-row, single-table
write — M2 and Log Groom each insert one `appointments` row, fully shaped. It
does **not** fully hold for a *dependent multi-row* write. Two hazards ride along
that a single-row dry-run never surfaces:

1. **Non-atomic inserts.** PostgREST has no multi-statement transaction — two
   separate `.insert()` calls mean a pet-insert failure *after* a client-insert
   success leaves an **orphan client**. A single-row surface has nothing to
   half-write; a dependent one does, and the dry-run cannot rehearse it.
2. **The gate lifting does not make the surface correct.** REQ-29 established
   that each surface gets its own write flag. REQ-30 adds: a dependent multi-row
   surface's flag must not be flippable on dry-run success *alone* — the part of
   the write the dry-run could not exercise (the FK wiring, the ordering, the
   rollback) still has to be built and verified.

**What Koya needs:** treat "dry-run passed" as a weaker signal for some surfaces.

1. Classify each dry-run write surface as **shape-fully-proven** (single row,
   single table) or **shape-partially-proven** (dependent multi-row). The
   per-surface write flag (REQ-29) should carry this classification.
2. A shape-partially-proven surface is **not cutover-eligible on dry-run success
   alone** — its checklist must include a verified atomic write path (a
   transactional RPC, or an explicit revert-on-failure of the parent row).
3. Its first-real-write verifier checks **both** rows landed *and* that forcing
   the child write to fail leaves **no orphan parent**.

**Relation to prior REQs:** REQ-28 is the dry-run pattern for the first write
surface; REQ-29 made per-surface write flags independent of the global security
gate. REQ-30 is the limit of the dry-run model itself — it cannot fully prove a
dependent multi-row write, so the platform must record which surfaces are only
partially proven and gate them on a transactional wrapper, not on dry-run
success.
