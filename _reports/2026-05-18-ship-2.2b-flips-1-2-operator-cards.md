# Ship 2.2b — write flips 1 & 2 operator cards

Execute-ready cards for the first two watch-week write flips: **Add Appointment**
then **Log Groom**. This is to `2026-05-18-ship-2.2b-write-gate-operations.md`
what the cutover operator card is to the runbook — the one-pager you keep open
while you flip. The write-gate-operations doc stays **authoritative** for flag
names, flip order, and rationale; nothing here changes a decision in it.

> **These cards do not authorize a flip.** Each flip is a separate, individually
> approved action. The live-write code already ships behind OFF-by-default
> flags; a flip = setting one Vercel env flag to `on` + redeploy. No code edit.

## Flip framework (recap)

- Flips happen **one at a time, across the ~7-day watch week**, smallest first:
  Add Appointment → Log Groom → Reminder → Add Household.
- Each surface is gated by a **server-only** env flag (not `NEXT_PUBLIC_`).
  Default OFF. A surface persists only when its flag is the exact string `on`.
- **Enable** = add the flag in Vercel Production + redeploy. **Disable** = remove
  it + redeploy (~40 s, no code revert). Disable is the watch-week kill path.
- A flip's mini-verifier must re-confirm the **not-yet-flipped** surfaces are
  still gated — no cascade (REQ-29).

## Appendix — the anon-write REST probe (shared)

Confirms the post-cutover RLS still rejects an unauthenticated write. Run from
any terminal. `ANON_KEY` is the **public** `NEXT_PUBLIC_SUPABASE_ANON_KEY` (the
same key in v1's HTML — not a secret, but pass it via env rather than pasting it
into a committed file):

```sh
export ANON_KEY='<the public anon key>'
curl -sS -o /dev/null -w 'HTTP %{http_code}\n' \
  -X POST 'https://pgkwovokciaqnbhpttba.supabase.co/rest/v1/appointments' \
  -H "apikey: $ANON_KEY" \
  -H "Authorization: Bearer $ANON_KEY" \
  -H 'Content-Type: application/json' \
  -d '{"client_id":"00000000-0000-0000-0000-000000000000","pet_id":"00000000-0000-0000-0000-000000000000","date":"2099-01-01"}'
unset ANON_KEY
```

- ✅ Pass: prints **`HTTP 401`** or **`HTTP 403`** — RLS rejected the write, so
  nothing was inserted.
- ❌ Fail: a `2xx` means an anon write succeeded — a serious RLS regression.
  Disable the just-flipped surface and treat it as a watch-week rollback trigger
  (runbook §4.1).

The probe is safe **because** the write is rejected: post-cutover the anon role
has no INSERT policy on `appointments`, so the row is never created.
