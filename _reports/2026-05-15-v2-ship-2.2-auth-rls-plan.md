# Tidy Tails v2 — Ship 2.2 Plan: Real Supabase Auth + RLS Hardening

## 0. How this relates to the design-lock spec

operationalizes its §3.2 (auth model), §3.3 (RLS model), §6 (data model), and §7
(migration) for the auth + RLS ship. It does not restate them.

If this plan and the design-lock spec conflict, the spec wins — unless a
superseding decision is logged in `docs/DECISIONS.md`. One such deviation is

## 1. Scope and decisions

### Ship 2.2 is two pieces

- **2.2a — App-side real auth.** Replace the Ship 2.1 placeholder cookie gate
  with real Supabase Auth: SSR Supabase client, email/password login, a real
  session check in `proxy.ts`. **Buildable now**, against the current
  (still-permissive) live database.
- **2.2b — The RLS-hardening migration.** Add `groomer_id` to all six public
  tables, backfill it, replace every permissive policy with `auth.uid()`-scoped
  policies, and drop the `client_overview` SECURITY DEFINER view. This is the
  schema change that closes **R-1** and **B2 / R-5**. It runs **once, as the
  cutover event** — not now.

### Decision logged 2026-05-15 (Russell): flag-day cutover

- The RLS migration runs as a single coordinated cutover event, not early in the
  ship sequence.
- v1 stays fully live and **code-unchanged** on its current permissive policies
  until the cutover moment.
- **No temporary anon bridge policies.** A bridge (permissive anon SELECT/INSERT/
  UPDATE kept alive past the migration) would keep the core R-1 hole open through
  the parallel-run week. Rejected. Not to be added unless Russell explicitly
  accepts that security tradeoff later.
- "Parallel run" is reinterpreted — see §2.
- **Consequence:** the migration is the *last* step of v2, after Ships 2.3–2.7
  make v2 Samantha's complete daily tool. 2.2a (app auth) can ship now; 2.2b (the
  migration) happens at cutover. The design-lock §10 ship order — which placed the
  RLS migration early — is re-derived by this decision.

### Deviation from design-lock spec §3.6 — ratified by Russell 2026-05-16

Spec §3.6 bundles **all** schema changes (auth/RLS + the `vaccinations` table +
`clients`/`pets` feature columns + optional `audit_events`) into "a single
Supabase migration." This plan **narrows the Ship 2.2 migration to security
only**: `groomer_id` + RLS policies + drop view. Feature schema ships as
**separate migrations**, in the ships that build those features (2.3+).

Rationale: a security migration with a small, well-understood surface has a far
simpler rollback than one also carrying feature columns and a new table. The
security migration and the feature migrations still all land before cutover.
**Ratified by Russell on 2026-05-16** — feature-schema migrations stay separate
from the security migration. Logged in `docs/DECISIONS.md` (supersedes spec §3.6
per spec §0).

### Auth provider — already locked, not re-opened

Supabase Auth, email/password (spec §3.1, §3.2, §11 — "auth provider already
decided"). No magic links; a forgotten-password email reset is sufficient for a
single operator.

## 2. The cutover event

Russell's specified six-step sequence, operationalized:

1. **Build + test v2 auth + RLS in dev/staging.** 2.2a app auth is built. The
   2.2b migration is authored as SQL and **rehearsed on a Supabase branch** (a
   throwaway copy of the project via `create_branch`), where the full verifier
   checklist (§7) is run green. v2 is feature-complete (Ships 2.3–2.7 done) and
   Samantha has done her pre-cutover rehearsal (below).
2. **Fresh backup.** Full logical dump immediately before the migration (§6),
   plus a captured snapshot of every current RLS policy and the `client_overview`
   view definition — these feed the rollback migration.
3. **Freeze v1 writes.** The freeze is atomic with the migration itself. The
   migration runs as a single transaction (§4); at the moment it commits, the
   scoped policies are in force and v1's anon key can no longer write — there is
   no half-migrated state v1 can observe. The only real exposure is a v1 write
   landing between the backup (step 2) and the migration's commit. Primary
   mitigation: take the backup in the seconds immediately before applying the
   migration. Second layer: coordinate a quiet window with Samantha (she confirms
   the day's appointments are logged) — v1 is static HTML with a hard-coded anon
   key and stays code-unchanged, so there is no technical freeze on the v1 side.
   The migration is small (runs in seconds) and atomic, so this window collapses
   to near-zero.
4. **Run schema + RLS migration.** The 2.2b migration (§4) applied to production
   via `apply_migration`. Scale is tiny (a few hundred clients/pets, well under
   a thousand appointments) — it runs in seconds.
5. **Deploy v2 as the live app.** v2 on Vercel becomes Samantha's tool at its
   production URL. Its authenticated Supabase client now reads her rows under the
   new scoped policies.
6. **v1 → archived fallback.** v1 stops being a live data client. See the
   v1-fallback note below.

### Reinterpreted "parallel run"

- **Pre-cutover rehearsal.** Before flag day, Samantha uses v2 on staging
  (read-only — fixtures or live SELECT) while **v1 remains her source of truth**.
  This is where she validates the v2 UX. No data divergence is possible because
  she is only *writing* in v1.
- **Post-cutover watch week.** After flag day, **v2 is the source of truth**. v1
  is reference/fallback only — not an active parallel-write surface. Watch ~7
  days against the design-lock §7.4 success criteria before declaring v1 fully
  decommissioned.

### v1-fallback honesty note

With no anon bridge (Russell's explicit instruction), v1's anon key is locked out
of **reads too** after step 4 — `groomer_id = auth.uid()` is false for the anon
role. So v1 cannot serve live data post-cutover. "v1 as fallback" therefore means
two concrete things, *not* a running read-only v1:

- (a) The **rollback migration** (§5) is the real recovery path — it restores the
  permissive state and v1 works again within minutes.
- (b) v1's HTML can be **archived** (branch-archived, or left at its URL) as a
  static reference to the old UI, but it will display no live data.
- If Russell wants v1 genuinely *viewable with data* during the watch week, that
  needs a one-time static data snapshot rendered into v1 — a small extra task.
  Flagged here, not assumed.

## 4. The RLS-hardening migration (Ship 2.2b — runs at cutover)

**Status: DRAFT SQL. Not run. No migration applied.**

### Pre-flight (read-only — must run first)

- `list_tables` + `list_migrations` + `get_advisors` (security) — snapshot the
  current schema, migration history, and advisor warnings.
- Capture the **exact definition of every existing policy** on the six tables
  (`SELECT * FROM pg_policies WHERE schemaname = 'public'`) and the
  **`client_overview` view definition** (`pg_get_viewdef`). Save to a snapshot
  file — the rollback migration (§5) is built from it.
- Confirm row counts via an exact `count(*)` per table and record them into the
  backup manifest (§6) — **that manifest is the one and only cutover baseline.**
  Do **not** treat any count printed in this plan, in `CLAUDE.md`, or in any
  `list_tables`/planner estimate as the baseline: those drift or estimate (a
  `list_tables` run on 2026-05-17 reported `appointments` ≈ 7 against a true
  737). The three empty tables (`booking_requests` / `client_accounts` /
  `automations_log`) should read 0. Full procedure:
  `_reports/2026-05-17-ship-2.2b-backup-rollback-rehearsal-checklist.md`.
- Confirm Samantha's `auth.users` row exists; capture her UID (prerequisite §8).

### Tables in scope (6)

`clients`, `pets`, `appointments`, `booking_requests`, `client_accounts`,
`automations_log`. All six are scoped — leaving a permissive policy on even an
empty table is still an open write hole, so R-1 is only fully closed by covering
all six.

`sam_review_responses` is left **exactly as-is** — its INSERT-only anon policy is
a Workstream-B artifact, not part of v2 (spec §3.3).

### Step A — add `groomer_id`, backfill, enforce `NOT NULL`

Per table (`:samantha_uid` is the literal UUID captured in pre-flight):

```sql
ALTER TABLE public.clients
  ADD COLUMN groomer_id uuid REFERENCES auth.users(id) DEFAULT auth.uid();
UPDATE public.clients SET groomer_id = :samantha_uid WHERE groomer_id IS NULL;
ALTER TABLE public.clients ALTER COLUMN groomer_id SET NOT NULL;
```

`DEFAULT auth.uid()` means future authenticated inserts auto-stamp the caller's
UID — multi-tenant-safe, trigger-free, and it satisfies the INSERT policy's
`WITH CHECK`. Existing rows are backfilled explicitly to Samantha. Repeat for all
six tables; the `UPDATE` is a harmless no-op on the three empty tables.

### Step B — replace permissive policies with scoped policies

Per table: drop **every** existing policy — exact names come from the pre-flight
snapshot; emit one `DROP POLICY IF EXISTS "<name>" ON public.clients;` per
captured policy (`IF EXISTS` keeps the migration idempotent on a re-run) — then
create the four scoped policies:

```sql
-- DROP POLICY IF EXISTS "<existing policy name>" ON public.clients;  -- one line per snapshot entry
ALTER TABLE public.clients ENABLE ROW LEVEL SECURITY;  -- already on; idempotent

CREATE POLICY "groomer_select" ON public.clients
  FOR SELECT USING (groomer_id = auth.uid());
CREATE POLICY "groomer_insert" ON public.clients
  FOR INSERT WITH CHECK (groomer_id = auth.uid());
CREATE POLICY "groomer_update" ON public.clients
  FOR UPDATE USING (groomer_id = auth.uid()) WITH CHECK (groomer_id = auth.uid());
CREATE POLICY "groomer_delete" ON public.clients
  FOR DELETE USING (groomer_id = auth.uid());
```

Two notes against spec §3.3:
- The UPDATE policy carries `WITH CHECK` as well as `USING`, so an update cannot
  reassign a row to a different `groomer_id`. Small hardening tweak.
- DELETE policies are (re-)created. v1 had them dropped 2026-04-22 as the R-1
  partial mitigation; under scoped RLS, controlled deletes for the authenticated
  groomer are safe, and v2 may need them.

Repeat for all six tables.

### Step C — drop the SECURITY DEFINER view

```sql
DROP VIEW IF EXISTS public.client_overview;
```

Closes B2 / R-5. The v2 scaffold does not query `client_overview` (grep-confirmed
2026-05-15); v1 may, but v1 is decommissioned in the same cutover event, so it is
moot. The view's query is rebuilt in the v2 app layer (spec §3.3).

### Apply

Steps A → B → C as **one named Supabase migration** via `apply_migration` — a
single revertible unit. Supabase migrations run inside one `BEGIN`/`COMMIT`
transaction, so A → B → C apply atomically: either the whole migration lands or
none of it does, and v1 is never left in a half-migrated state (this is what
makes the §2 step-3 write-freeze atomic). `:samantha_uid` is a placeholder, not a
bind parameter — `apply_migration` has no parameter binding for DDL, so the
literal UUID captured in pre-flight is substituted into the SQL text before the
migration is applied.

**Forward-dependency note:** because `groomer_id`'s default is `auth.uid()`, the
Ship 2.4 write actions do not strictly need to set `groomer_id` — but they should
set it explicitly for clarity, or the team consciously relies on the default.
Recorded so Ship 2.4 planning picks it up.

## 5. Rollback plan

The forward migration is one named Supabase migration → rollback is its inverse,
authored from the pre-flight snapshot and **rehearsed green on a Supabase branch
before flag day**.

### Rollback migration (inverse of §4)

- **Step C⁻¹** — recreate `client_overview` from the captured view definition.
- **Step B⁻¹** — drop the `groomer_*` scoped policies; recreate the original
  permissive policies from the snapshot (`qual = true`, `roles = {public}` for
  SELECT/INSERT/UPDATE; DELETE stays dropped, matching the 2026-04-22 state).
- **Step A⁻¹** — `ALTER TABLE ... DROP COLUMN groomer_id` on all six tables.

### Data safety

Any rows v2 creates between the migration and a rollback are **preserved** —
rollback only drops the `groomer_id` column, not rows. v1 reads all columns, so
it sees those rows fine after a rollback.

### When rollback is the move

During the post-cutover watch week, if v2 has a blocking defect, applying the
rollback migration restores the permissive state and v1 becomes a working data
client again within minutes. **This is the real "v1 fallback"** (see §2). v2
build then resumes from a known-good state.

### Two independent safety nets

1. The rehearsed rollback migration (fast, in-place).
2. The fresh logical backup (§6) for a full restore if a rollback somehow cannot
   be applied.

Plus the Supabase-branch rehearsal means the forward migration is **never
first-run in production**.

### The cutover backup itself

- Taken **immediately before** the migration (step 2 of §2) — not a stale dump.
- Full schema + data for all six tables (`clients` / `pets` / `appointments` plus
  the three empty tables), plus the pre-flight policy/view snapshot.

  repo** — never committed; it is real customer data.

dump — and removes the free-tier inactivity-pause risk (R-2). Spec §3.5
recommends it "at or before V1_HARDENED." Recommend enabling it before flag day.
Flagged as a Russell cost decision; not a hard blocker if the logical dump plus
the branch rehearsal are in place.

## 7. Verifier checklist

Run the whole checklist on the **Supabase branch rehearsal first**, then again on
**production at cutover**. Definition of done for R-1 closure (design-lock §10):
an anonymous REST request returns *empty results* — no data, and not an error.

### Pre-flight (before the migration)

- [ ] Current RLS policy definitions for all six tables captured to a snapshot file.
- [ ] `client_overview` view definition captured.
- [ ] Row counts recorded via exact `count(*)` into the backup manifest (the cutover baseline); the three empty tables read 0.
- [ ] Samantha's `auth.users` row exists; her UID captured for the backfill.
- [ ] Fresh full backup taken and verified readable (§6).
- [ ] Rollback migration authored from the snapshot and rehearsed on a Supabase branch. "Rehearsed green" means, after applying forward-then-rollback on the branch:
  - [ ] Row counts match the pre-rehearsal baseline captured in the backup manifest.
  - [ ] `client_overview` exists again, with the captured definition.
  - [ ] `groomer_id` is absent from all six tables.
  - [ ] The original permissive policies are back in place (`qual = true`, `roles = {public}`).
  - [ ] An anonymous REST request returns rows again — proving v1 would work after a real rollback (the inverse of the R-1-closed test).
- [ ] Forward migration run green on the same branch with this checklist passing.
- [ ] v1 write-freeze window coordinated with Samantha.

### Post-migration — database

- [ ] **R-1 closed:** an anonymous request (anon key, no session) to `/rest/v1/clients` returns `[]` — empty, HTTP 200, not 401/403. Repeat for `pets` and `appointments`.
- [ ] An anonymous INSERT / UPDATE to any of the six tables is rejected (RLS `WITH CHECK` fails).
- [ ] An authenticated request as Samantha returns her rows; counts still match the backup-manifest baseline.
- [ ] `groomer_id` is `NOT NULL` on all six tables; every existing row equals Samantha's UID.
- [ ] No policy with `qual = true` / `roles = {public}` remains on any of the six tables (`SELECT * FROM pg_policies`).
- [ ] The `client_overview` view no longer exists.
- [ ] `get_advisors` (security): the permissive-policy / R-1 findings and the `client_overview` SECURITY DEFINER warning are cleared.

### Post-migration — app

- [ ] Login with Samantha's real email/password succeeds; a wrong password is rejected with a clear message.
- [ ] Session persists across a page refresh; an expired session lands on `/login`, not a broken page.
- [ ] Sign-out clears the session and returns to `/login`.
- [ ] `proxy.ts` redirects an unauthenticated request to `/login`.
- [ ] Authenticated v2 loads the client list, client detail, pet detail, and reports with real data; counts/figures match.
- [ ] Revenue total spot-checked against the design-lock §4.6 figure (~$57,821.25). That figure is design-lock-era — the appointment set has since grown past the `730` quoted there; re-derive against the live data rather than reusing `730` as a current count.

- [ ] v2 deployed and reachable at its production URL.
- [ ] v1 confirmed no longer serving live data (expected — confirms the anon lockout). v1 archived per §2.
- [ ] Watch-week monitoring in place — concretely: Vercel error logs checked daily, `get_advisors` (security) re-run at least once during the week, `get_logs` scanned for auth/postgres anomalies, and a daily check-in with Samantha on any workflow break. Design-lock §7.4 success criteria tracked.

## 8. Prerequisites and what this plan does NOT do

### Hard prerequisites before the 2.2b migration runs (all blockers)

1. **Samantha's `auth.users` account created** (Russell, via the Supabase
   dashboard or a one-time signup) and her UID captured. `groomer_id NOT NULL FK
   → auth.users` cannot be backfilled without it. **This is Step 0.**
2. **`dump_supabase.py` written, or the REST-API backup procedure ratified** (§6).
3. **v2 feature-complete** — Ships 2.3–2.7 done — because flag day decommissions
   v1; v2 must be Samantha's full daily tool first.
4. **Rollback migration authored and rehearsed green on a Supabase branch** (§5).
5. **Fresh backup** taken immediately before the migration (§6).

### Not a blocker (decoupled)

Contact-card reconciliation Phases 2–4. Spec §3.6 step 1 wants the DB clean for
v2 *launch quality*, but the migration is schema/policy-only — ghost rows just
receive `groomer_id` like any row and do not affect RLS. Reconciliation and Ship
2.2b can proceed independently; only the cutover *quality bar* links them.

### This plan does NOT

- Run any SQL, apply any migration, or change any schema. It is a plan.
- Create Samantha's auth account or take a backup — those are prerequisite
  actions for Russell to authorize and perform.
- Cover the feature-schema migration (`vaccinations` table, `clients`/`pets`
  feature columns, optional `audit_events`) — deferred to the feature ships (§1
  deviation).
- Set a date. Flag day is milestone-gated on the five prerequisites above, per
  `.koya/SOUL.md` (milestones, not time).
