# Ship 2.2b — Backup + Rollback-Rehearsal Operator Checklist

This is the **operator runbook** for the RLS cutover. It turns the two parent
plans into a sequence of checkboxes with explicit pass/fail gates.

- The **migration SQL** is authored in `2026-05-15-v2-ship-2.2-auth-rls-plan.md`
  §4 (forward) and §5 (rollback). This checklist **references** it — it does not
  re-author or restate it.
- The **why / scope / decisions** live in the two parent plans. This document is
  the *how* and the *in-what-order*.

**Nothing here has been executed.** Working through it is a separate, explicit
action Russell authorizes step by step.

---

## §0 — What this checklist does NOT authorize

Reading or approving this document does **not** authorize any of: running the
migration, running any SQL against production, mutating Supabase, lifting any
write gate, or sending any SMS. Each phase below is its own go/no-go. The
cutover (§F) and each surface flip (§G) need Russell's explicit, separate
authorization at the moment they happen.

## §A — Pre-cutover backup requirements

### A.1 Scope — back up all 7 public tables

The migration scopes **6** tables; the disaster-recovery backup covers **all 7**
public tables (a backup is comprehensive, not migration-scoped):

| Table | Migration-scoped? | In backup? |
|---|---|---|
| `clients` | ✅ | ✅ |
| `pets` | ✅ | ✅ |
| `appointments` | ✅ | ✅ |
| `booking_requests` | ✅ | ✅ |
| `client_accounts` | ✅ | ✅ |
| `automations_log` | ✅ | ✅ |
| `sam_review_responses` | ❌ left as-is (Workstream-B artifact) | ✅ — 69 rows of real data, still backed up |

For each table the backup captures **schema (DDL) + every data row**. Output
format mirrors the existing `~/venture-ops/backups/tidy-tails/` precedent: a
combined schema+data SQL dump **plus** per-table CSVs.

### A.2 `auth` schema / `auth.users` consideration

The migration adds `groomer_id uuid REFERENCES auth.users(id)`. `auth.users` is
**not** part of a `public`-schema logical dump. Three concrete rules:

- [ ] Capture Samantha's `auth.users` **UID + email** into the manifest at
  pre-flight (a UID is not a secret; it is the literal substituted into the
  migration's backfill and asserted by the verifier).
- [ ] Do **not** dump or restore `auth.users` via `dump_supabase.py` — it is
  Supabase-Auth-managed, contains password hashes, and is outside the script's
  contract. It is covered by Supabase's platform backups (and PITR if Pro is on).
- [ ] Rollback does **not** need `auth.users` restored — rollback drops the
  `groomer_id` FK column, so `auth.users` is not on the rollback critical path.
  (Watch-week rule: do not delete/recreate Samantha's auth user — see §H.)

### A.3 Manifest — the cutover baseline

Every backup writes a `manifest.json` (or `.md`). The manifest **is the cutover
baseline** — the verifier compares against it, never against a count written
into prose:

- [ ] UTC timestamp; Supabase project ref (`pgkwovokciaqnbhpttba`).
- [ ] Per-table **exact** row count — `count(*)`, never a planner estimate
  (see Appendix on why `list_tables` counts are not trustworthy).
- [ ] Per-file **SHA-256**.
- [ ] Dump method used (script vs ratified REST procedure) and tool version.
- [ ] Supabase migration-history head at backup time.
- [ ] Samantha's auth UID + email.
- [ ] Operator name + a note that this manifest's counts are the authoritative
  forward/rollback/post-cutover baseline.

### A.4 Where the backup lives

- [ ] Stored at `~/venture-ops/backups/tidy-tails/<cutover-date>/` — **outside
  this repo** (confirm the canonical path against CLAUDE.md before running).
- [ ] **Never committed.** It is real customer data. Confirm repo hygiene
  excludes it.

### A.5 Verify the backup can be read back

"Backup taken" is not "backup verified." A backup is verified only when:

- [ ] Every file's SHA-256 re-checked and matched against the manifest.
- [ ] The dump re-imported into a **throwaway** target (local Postgres or a
  Supabase branch) and per-table row counts match the manifest **exactly**.
- [ ] A known record spot-checked in the readback (e.g. a named client).

### A.6 Freshness

- [ ] A **rehearsal** backup may be taken earlier (to drive §D/§E).
- [ ] The **cutover** backup is taken in the minutes immediately before the
  migration (§F) — a stale dump does not count.

## §B — `dump_supabase.py` requirements

Both parent plans record that `dump_supabase.py` **does not exist** and the
2026-05-15 backup used a REST-API workaround. Before cutover, **pick one**:

- [ ] **(Recommended)** Write `venture-ops/dump_supabase.py`.
- [ ] **Or** formally ratify the REST-API workaround as the documented official
  backup procedure, recording the exact steps.

### B.1 Where it lives

- [ ] `venture-ops/dump_supabase.py` — in the **studio** repo, referenced
  studio-wide. **Not** committed into the `tidy-tails` repo.

### B.2 What it must export

- [ ] All `public`-schema tables of a given project: schema DDL + every data row.
- [ ] Output: per-table CSV **and** a combined schema+data SQL dump.
- [ ] The manifest (§A.3).

### B.3 How it must paginate

- [ ] Page every table in fixed-size ranges (the PostgREST/Supabase ~1000-row
  cap) — the same `fetchAllRows` pattern v2 already uses in `lib/data/live.ts`.
- [ ] A table larger than the cap must **never silently truncate**
  (`appointments` is under 1000 today but will cross it).
- [ ] After dumping each table, reconcile: exported row count **must equal** the
  server `count(*)`. Mismatch → fail loudly, non-zero exit.

### B.4 Metadata / manifest

- [ ] Writes the §A.3 manifest, including the exported-vs-server count
  reconciliation result per table.

### B.5 Credentials — explicit no-service-role-in-repo rule

- [ ] Reads the Supabase URL + key from **environment variables or a local
  secrets file outside any git-tracked directory** — never hardcoded.
- [ ] Uses the **service_role** key (to bypass RLS and guarantee a complete
  read — essential post-cutover when the anon key sees nothing).
- [ ] **The service_role key never lands anywhere persisted:** not in the repo,
  not in the script, not in the manifest, not in the dump files, not in stdout
  or logs. The script refuses to start if the credential env var is missing, and
  self-redacts the key in any error message.
- [ ] No connection string containing a DB password is committed or logged.
- [ ] Re-running produces a new dated directory; it never overwrites a prior dump.

## §C — Policy / view / schema snapshot (rollback raw material)

Captured **read-only** at pre-flight. This snapshot is what the rollback
migration (§E) is authored from. Policy/view/schema definitions contain **no
customer data** — they are safe to keep as dated `_reports/` artifacts for
review; row data is not.

- [ ] **RLS policies** — capture the full definition of every policy on all 6
  (7) tables (`SELECT * FROM pg_policies WHERE schemaname='public'`): name, cmd,
  roles, `qual`, `with_check`. Save verbatim.
- [ ] **`client_overview` view** — capture the exact definition
  (`pg_get_viewdef('public.client_overview', true)`) **before** the migration
  drops it.
- [ ] **Table schemas** — capture the full DDL / column definitions of all 6
  tables and their RLS-enabled state (`list_tables` verbose / schema-only dump),
  so the pre-migration shape is on record.
- [ ] **Security advisors** — capture `get_advisors` (security) output as the
  *before* state, so post-cutover the cleared findings can be proven.
- [ ] Save as dated artifacts: `<cutover-date>-preflight-policies.sql`,
  `-client_overview.sql`, `-schema.sql`, `-advisors.json` — in the backup
  directory **and** as repo `_reports/` artifacts (schema/policy only — no data).

## §D — Forward migration rehearsal (Supabase branch — never production)

The forward migration is **never first-run in production**. Rehearse it on a
Supabase branch (`create_branch` — a throwaway copy) or a throwaway clone.

- [ ] Create a Supabase branch from production.
- [ ] **Confirm the branch carries data.** Branches copy schema; whether they
  carry data depends on branch type. If the branch is empty, seed it from the
  backup (§A) — otherwise the backfill and count verifier pass *vacuously*
  against an empty table and prove nothing.
- [ ] Apply the Ship 2.2b forward migration (parent plan §4, with
  `:samantha_uid` substituted) to the branch via `apply_migration`.
- [ ] **Verify `groomer_id`:** column added to all 6 tables; backfilled (zero
  NULLs); `NOT NULL` enforced; every row equals Samantha's UID; `DEFAULT
  auth.uid()` present.
- [ ] **Verify scoped policies:** each of the 6 tables has exactly the 4
  `groomer_*` policies; **no** policy with `qual=true` / `roles={public}`
  remains (`pg_policies` confirms).
- [ ] **Verify `client_overview` dropped.**
- [ ] **Verify v2 authenticated reads still work:** point a v2 instance (or run a
  direct authenticated query as Samantha) at the branch — client list, client
  detail, pet detail, and reports load; counts match the manifest baseline;
  revenue total spot-checked.
- [ ] **Verify anon is rejected:** anon-key REST `GET` on `clients` / `pets` /
  `appointments` returns `[]` (HTTP 200, empty — the R-1-closed definition);
  anon `INSERT` / `UPDATE` / `DELETE` rejected.
- [ ] Run the full parent-plan §7 verifier on the branch.
- [ ] Record the rehearsal result as a dated artifact (what ran, pass/fail per
  gate).

**"Forward rehearsal green"** = every parent-plan §7 gate passes on the branch.
Keep the branch — §E rehearses rollback on the same one.

## §E — Rollback rehearsal

- [ ] Author the **inverse migration** from the §C snapshot (parent plan §5):
  - C⁻¹ — recreate `client_overview` from the captured view definition.
  - B⁻¹ — drop the `groomer_*` policies; recreate the captured permissive
    policies (DELETE stays dropped, matching the 2026-04-22 state).
  - A⁻¹ — `DROP COLUMN groomer_id` on all 6 tables.
- [ ] On the **same branch**, immediately after the forward rehearsal: apply
  forward → then apply rollback.
- [ ] Save the rollback SQL as a dated artifact alongside the forward migration,
  for Russell to review — ready to paste at cutover if needed.

### "Rollback rehearsed green" — exact definition

After forward-then-rollback on the branch, **all** of the following hold:

- [ ] Per-table row counts match the pre-rehearsal manifest baseline **exactly**.
- [ ] `client_overview` exists again — byte-compare its definition against the
  §C capture.
- [ ] `groomer_id` is absent from all 6 tables.
- [ ] The original permissive policies are back (`qual=true`, `roles={public}`
  for SELECT/INSERT/UPDATE; DELETE stays dropped).
- [ ] An anon REST request returns **rows again** — proving v1 would work after a
  real rollback (the inverse of the R-1-closed test).
- [ ] `get_advisors` (security) shows the original pre-migration finding set —
  the state is genuinely restored, not merely "no error."
- [ ] **Data-preservation check:** create a test row on the branch between the
  forward and rollback steps; confirm it survives the rollback (rollback drops a
  column, not rows).

### Two independent safety nets (restated)

1. The rehearsed rollback migration — fast, in-place.
2. The fresh logical backup (§A) — a full restore if the rollback itself somehow
   cannot be applied.

## §F — Production cutover runbook

Each step is its own checkbox. Stop and reassess on any failed gate.

### F.1 Pre-flight — all hard prerequisites confirmed

- [ ] Samantha's `auth.users` UID captured (account exists).
- [ ] `dump_supabase.py` written, **or** the REST backup procedure ratified (§B).
- [ ] v2 covers Samantha's full daily workflow — search ✅, Add Appointment ✅,
  Log Groom ✅, Reminder prep ✅, Add Household intake ✅ (all built dry-run/
  gated and committed this session).
- [ ] Forward rehearsal green on a Supabase branch (§D).
- [ ] Rollback rehearsed green on a Supabase branch (§E).
- [ ] Supabase Pro / PITR decision made (recommended before flag day — a Russell
  cost decision; not a hard blocker if the logical dump + branch rehearsal hold).
- [ ] The §4.1 cutover-scope narrowing logged in `docs/DECISIONS.md`.

### F.2 Quiet window + final backup

- [ ] Coordinate a ~30-minute quiet window with Samantha — she confirms the
  day's appointments / grooms are logged in v1.
- [ ] Take the **final fresh backup** (§A) — readback-verified, manifest pinned.
  **This manifest's exact counts are THE cutover baseline** for every gate below.
- [ ] Re-capture the §C snapshot on production one more time — confirm nothing
  drifted (policies / view / advisors) since the rehearsal.
- [ ] Re-confirm Samantha's UID; substitute it into the migration SQL.

### F.3 Apply

- [ ] **Warn Russell explicitly:** applying the migration takes v1 dark — v1's
  anon key loses **reads and writes** the instant it commits. This is the point
  of no easy return; rollback (§E) is the recovery path.
- [ ] Apply the Ship 2.2b migration to production — one named migration, one
  transaction (parent plan §4).

### F.4 Post-migration verifier (production)

- [ ] Row counts unchanged vs the **manifest baseline** (all 6 tables).
- [ ] `groomer_id` `NOT NULL` and equal to Samantha's UID on every row of all 6.
- [ ] Only `groomer_*` scoped policies remain — no `qual=true`/`public` policy.
- [ ] `client_overview` no longer exists.
- [ ] `get_advisors` (security): R-1 / permissive-policy / SECURITY DEFINER
  findings cleared.
- [ ] v2 login succeeds; authenticated reads load (client list / detail / pet /
  reports); revenue spot-checked.
- [ ] Anon REST blocked: anon `GET` clients/pets/appointments → `[]`; anon write
  rejected.
- [ ] **Writes still gated** — confirm all 4 write surfaces (Add Appointment,
  Log Groom, Reminder send, Add Household) **still return their gated result**.
  The migration makes writes *safe*, not *active*; no flag is flipped yet. An
  operator must not assume "migration committed → writes work" — they do not
  until §G.

### F.5 Cutover state

- [ ] v2 deployed and reachable at its production URL; env vars confirmed.
- [ ] v1 confirmed no longer serving live data (expected — confirms the anon
  lockout); v1 archived per parent-plan §2 step 6.
- [ ] Cutover logged in `docs/DECISIONS.md`.

### F.6 Abort path

- [ ] If **any** F.4 gate fails → STOP. Apply the rehearsed rollback (§E),
  return to the known-good permissive state, do **not** proceed to §G.

## §G — Per-surface post-cutover write-gate flip order

Each flip = **one code change + one `docs/DECISIONS.md` entry + one
mini-verifier**. Flipped **one at a time**, spaced across the watch week — never
all on day one, never a big-bang. Every mini-verifier also re-checks that the
**not-yet-flipped surfaces are still gated** (a flip must not cascade — REQ-29).

### G.1 — Add Appointment (first)

Smallest, best-rehearsed surface (M2, committed `dbbc2f2`).

- [ ] Flip: add `groomer_id` to the `buildAppointmentInsert` payload (explicit);
  replace the live `gated` branch in `createBooking` with `.insert()` +
  `revalidatePath`; add a `saved` result state distinct from the dry-run `demo`.
- [ ] Mini-verifier: book one test appointment → exactly **one** `appointments`
  row added → its `groomer_id` equals Samantha's UID → live count moves
  `baseline → baseline + 1` → an anon INSERT to `appointments` is still rejected
  at that moment → Log Groom / Reminder / Add Household **still gated**.

### G.2 — Log Groom (second)

- [ ] Flip: same shape — `.insert()` with `status='completed'` + explicit
  `groomer_id`.
- [ ] Mini-verifier: log one test groom → exactly **one** row with
  `status='completed'`, scoped to Samantha → anon write still rejected →
  Reminder / Add Household **still gated**.

### G.3 — Reminder send (third) — still human-confirmed per message

- [ ] Flip: wire the confirmed-send path to the `send-sms` Twilio edge function
  (confirm its request contract first); write one `automations_log` row per
  send.
- [ ] **REQ-31 invariant:** the flag authorizes the *capability* — never an
  automatic, batch, scheduled, or background send. Every SMS still requires a
  fresh per-message confirmation tap from Samantha.
- [ ] Mini-verifier: a test SMS goes **only** on an explicit per-message confirm
  → no message sends without it → exactly **one** `automations_log` row appears,
  scoped to Samantha → no scheduler/batch code path exists → Add Household
  **still gated**.

### G.4 — Add Household (last) — blocked on the transactional wrapper

- [ ] **Precondition (REQ-30):** the live path is a `clients` insert + a
  dependent `pets` insert; the two are **not atomic** in PostgREST — a pet-insert
  failure after a client-insert success orphans a client. This flip is **blocked
  until** a transactional wrapper (a Postgres RPC, or an explicit
  revert-on-failure of the client row) is built and itself verified.
- [ ] Flip: route the live insert through that wrapper; both rows carry
  `groomer_id`.
- [ ] Mini-verifier: create one test household → exactly **one** `clients` row +
  **one** `pets` row, both scoped to Samantha, `pets.client_id` correct →
  forcing the pet insert to fail leaves **no orphan client** → anon write
  rejected.

## §H — Watch-week plan (~7 days)

After cutover, v2 is the source of truth; v1 is archived reference only. Watch
~7 days against the design-lock §7.4 success criteria before declaring v1
decommissioned.

### H.1 Monitor daily

- [ ] Vercel error logs checked.
- [ ] Supabase `get_logs` scanned for auth / postgres anomalies.
- [ ] `get_advisors` (security) re-run at least once mid-week — R-1 / SECURITY
  DEFINER findings confirmed still cleared.
- [ ] Row counts sanity-checked vs expected growth — only flipped surfaces add
  rows, and the delta should match Samantha's actual activity.
- [ ] Daily check-in with Samantha.

### H.2 What Samantha reports

- Any workflow slower or broken vs v1.
- Any client / pet / appointment that looks wrong, missing, or mis-scoped.
- Any reminder that did **not** send when she confirmed it.
- **Critically:** any text that went out she did **not** confirm.
- Any login / session trouble.

### H.3 Rollback criteria — apply the rehearsed rollback if ANY hold

- [ ] A blocking defect prevents Samantha's core daily work (look up, book, log,
  remind) and there is no same-day fix.
- [ ] Data integrity is in doubt — rows wrong, missing, or mis-scoped.
- [ ] **Any unconfirmed SMS goes out** — a hard-rule (REQ-31) violation, its own
  standalone tripwire.

Rollback restores the permissive state; v1 works again within minutes; v2 build
resumes from a known-good state.

### H.4 What NOT to change during the watch week

- No schema changes, no new migrations, no feature ships.
- No RLS policy edits.
- No write-surface flips beyond the planned 4 — and those one at a time, each
  mini-verified before the next.
- No data-lane reconciliation that touches the 6 tables.
- The week is for **observation**. The only changes permitted are the planned
  per-surface flips (§G) and an emergency rollback (§H.3).

### H.5 Exit

- [ ] After ~7 clean days against the design-lock §7.4 success criteria →
  declare v1 decommissioned, watch week closed; log in `docs/DECISIONS.md`.

## Appendix — row counts: read this before setting any verifier baseline

**Every row count written into a planning document is stale or estimated. Do
not use one as a verifier baseline.**

| Source | clients | pets | appointments |
|---|---|---|---|
| CLAUDE.md / Ship 2.2 plan §7 | 268 | 352 | 730 |
| Production-mode plan §4.5 | 268 | 352 | 737 |
| `list_tables` planner estimate (this session) | — | — | **7** ⚠️ |
| Browser-pass verified counts (this session) | 137 | 188 | 737 |

The data was reconciled (contact-card Phases 1–3.5), so the older `268 / 352`
counts are obsolete. And `list_tables` reports `pg_class.reltuples` — a **planner
estimate**, not a count; this session it returned `appointments: 7` against a
true `737`. A verifier built on either a prose number or a planner estimate
could "pass" against a destroyed database.

**The rule:** the one and only cutover baseline is the **exact `count(*)`
captured into the backup manifest at backup time** (§A.3). Every gate in §D, §E,
§F, and §G compares against that manifest — never against a number in any
planning doc, including this one. Current observed (2026-05-17) counts —
`clients` 137, `pets` 188, `appointments` 737, `booking_requests` /
`client_accounts` / `automations_log` 0, `sam_review_responses` 69 — are recorded
here only as a sanity reference, **not** as a baseline.
