## The flag mechanism (recap)

Each flip adds the real `.insert()` path **behind a per-surface env flag**
(parent plan). Flag unset/`off` → the action still returns `gated`. Flag `on`
→ it persists. **Enable** = set the flag in Vercel Production env + redeploy.
**Disable** = unset it + redeploy — no code revert. The flag is read in the
server action via `process.env`.

> **Precondition for Surface 1 — fix `lastAppointment()` first.** Per the
> doc-gaps note, `v2/lib/derive.ts` `lastAppointment()` has no past/future
> filter; once a future-dated `appointments` row exists (which an Add
> Appointment booking is), "last visit" displays a future date. The Add
> Appointment flip is the first thing that creates such rows, so this small fix
> is a **precondition of the Surface 1 flip**, not of the cutover. Schedule it
> into the Surface 1 flip work.

**1b. Imports** — add at the top of `appointments.ts`:

```ts

```

**1c. The live branch** — replace the current `gated` return
(`appointments.ts:116-122`, the `return { status: "gated", … }` block) with:

```ts
  // Live mode. The per-surface flag decides: off → still gated; on → persist.
  if (process.env.TIDYTAILS_ENABLE_ADD_APPOINTMENT_WRITE !== "on") {
    return {
      status: "gated",
      summary,
      message:
        "Live booking writes aren't switched on yet — they turn on after the " +
        "Ship 2.2b security cutover. Nothing was saved.",
    };
  }

  const supabase = await createServerSupabase();
  const { error } = await supabase.from("appointments").insert(payload);
  if (error) {
    return {
      status: "error",
      errors: {},
      formError: "That appointment could not be saved. Nothing was written.",
    };
  }
  revalidatePath(`/clients/${booking.client_id}`);
  return { status: "saved", summary };
```

`payload` carries no `groomer_id` — the column `DEFAULT auth.uid()` stamps it
to Samantha on her authenticated insert (proven in the branch rehearsal). The
insert is a single row, so an `error` means nothing was partially written.

**Expected row-count delta:** `appointments` **+1** per booking; all other
tables unchanged. The new row's `groomer_id` = `88413167-0799-49a7-ba4c-c1c29403e038`.

**Mini-verifier:** book one real test appointment → exactly **one** new
`appointments` row → its `groomer_id` = Samantha's UID → live count moves
`baseline → +1` → an anon REST `INSERT` to `appointments` is still rejected
`401/403` → Log Groom / Reminder / Add Household **still return `gated`** (no
cascade — REQ-29).

**Disable / rollback:** unset `TIDYTAILS_ENABLE_ADD_APPOINTMENT_WRITE` in Vercel +
redeploy → the action returns to `gated`. If a bad row was written, delete it
as Samantha via the authenticated client (the `groomer_delete` policy permits
her own-row deletes). This surface does not, on its own, warrant an RLS rollback.

## 3. Surfaces 3 & 4 — not code-prepped here

These stay exactly as the parent write-flip plan describes; they are **not**
detailed here because each needs work the parent plan already scopes:

- **Surface 3 — Reminder send.** The parent plan's flip *adds* a new
  `confirmReminderSend` server action — `reminders.ts` today only *prepares* a
  draft, there is no send action to un-gate. The send path is **human-confirmed
  forever**: callable only from an explicit per-message confirm control, never
  batched, scheduled, or automatic (REQ-31, HARD rule). It requires the
  Twilio-backed `send-sms` edge function (its secrets live in Supabase function
  secrets, not the repo) and a `NEXT_PUBLIC_WRITES_REMINDER` flag. Code-prep it
  in its own pass when Surface 2 is verified and Russell approves Surface 3.
- **Surface 4 — Add Household.** Blocked on a transactional wrapper (REQ-30): a
  `clients` insert plus a dependent `pets` insert are not atomic in PostgREST, so
  a pet-insert failure orphans a client row. The fix is a Postgres RPC
  (`SECURITY INVOKER`, inserts both rows in one transaction) — itself a schema
  change needing its own migration, branch rehearsal, and approval, i.e. a
  **separate small ship**. Do not flip Surface 4 until that RPC exists and is
  verified.
