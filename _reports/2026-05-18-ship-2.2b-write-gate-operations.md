# Ship 2.2b ‚Äî write-gate operations

This is the **authoritative** reference for v2's write kill-switches: the flag
names, their defaults, the flip order, and exactly how to enable, verify, and
disable each write surface. The strategy and rationale are in the write-flip
plan; this doc is the operating procedure.

## The write-gate framework

Each of v2's four write surfaces is gated by a **private, server-only**
environment flag, read in the server action via `v2/lib/writeGate.ts`. The flag
names are deliberately **not** prefixed `NEXT_PUBLIC_` ‚Äî their values must never
reach the browser bundle. Default is **OFF**; a surface persists data only when
its flag is the **exact string `on`** (unset, empty, `false`, `off`, `true`,
`ON`, ` on `, or anything else all read as OFF ‚Äî fail-safe).

| Surface | Env flag | Gate function | Default | Implemented? |
|---|---|---|---|---|
| Add Appointment | `TIDYTAILS_ENABLE_ADD_APPOINTMENT_WRITE` | `isAddAppointmentWriteEnabled()` | OFF | ‚úÖ live-write path implemented behind the flag |
| Log Groom | `TIDYTAILS_ENABLE_LOG_GROOM_WRITE` | `isLogGroomWriteEnabled()` | OFF | ‚úÖ live-write path implemented behind the flag |
| Reminder send | `TIDYTAILS_ENABLE_REMINDER_SEND` | `isReminderSendEnabled()` | OFF | ‚è≥ flag exists; the confirmed-send action is not built yet |
| Add Household | `TIDYTAILS_ENABLE_ADD_HOUSEHOLD_WRITE` | `isAddHouseholdWriteEnabled()` | OFF | ‚è≥ flag exists; needs the transactional RPC (see the RPC plan) |

**No flag is set in any environment.** All four surfaces are OFF in Vercel
Production today. Enabling one is a deliberate, separately-approved act.

## Flip order ‚Äî never a big-bang

Flips happen **one at a time, across the ~7-day watch week**, in
operator-confidence order (smallest/safest first):

1. **Add Appointment** ‚Äî trigger: the Ship 2.2b cutover is green.
2. **Log Groom** ‚Äî trigger: Add Appointment flipped + mini-verified + approved.
3. **Reminder send** ‚Äî trigger: Log Groom flipped + mini-verified + approved;
   *and* the confirmed-send action is built (it does not exist yet).
4. **Add Household** ‚Äî trigger: Reminder flipped + mini-verified + approved;
   *and* the transactional RPC is rehearsed and applied (see
   `2026-05-18-add-household-rpc-plan.md`).

Each flip needs its own explicit Russell approval. A flip's mini-verifier must
re-confirm the not-yet-flipped surfaces are **still gated** (no cascade).

## How to enable a surface

Enabling is a Vercel env change + redeploy ‚Äî never a code edit.

```sh
# From the v2/ directory. Example: Add Appointment.
vercel env add TIDYTAILS_ENABLE_ADD_APPOINTMENT_WRITE production
#   ‚Üí when prompted for the value, enter exactly:  on
vercel --prod                 # redeploy so the new env value takes effect
```

(Or set it in the Vercel dashboard ‚Üí project `tidy-tails-v2` ‚Üí Settings ‚Üí
Environment Variables ‚Üí Production, then redeploy.)

## How to disable a surface ‚Äî fast

```sh
# From the v2/ directory. Example: Add Appointment.
vercel env rm TIDYTAILS_ENABLE_ADD_APPOINTMENT_WRITE production
vercel --prod                 # redeploy ‚Äî the action returns to `gated`
```

Removing the flag (or setting it to anything other than `on`) + redeploying
returns the surface to its gated state in the time of one deploy (~40s). No
code revert. This is the watch-week kill path.

## Per-surface flip procedure

### 1. Add Appointment

**Precondition:** Ship 2.2b cutover green; the `lastAppointment()` future-date
fix done (`derive.ts` ‚Äî see the doc-gaps note; a booking creates the first
future-dated row, which is what makes that bug active).

**Enable:** set `TIDYTAILS_ENABLE_ADD_APPOINTMENT_WRITE=on` (Production) +
`vercel --prod`.

**Verify (mini-verifier):**
- Book one real test appointment in v2 ‚Üí exactly **one** new `appointments` row.
- Its `groomer_id` = Samantha's UID (`88413167-0799-49a7-ba4c-c1c29403e038`).
- Live `appointments` count moves `baseline ‚Üí +1` (baseline = the cutover
  `MANIFEST.json`).
- An anon REST `INSERT` to `appointments` is still rejected `401/403`.
- Log Groom, Reminder, Add Household **still return `gated`**.

**Disable:** `vercel env rm TIDYTAILS_ENABLE_ADD_APPOINTMENT_WRITE production` +
`vercel --prod`. If a bad row was written, delete it as Samantha via the
authenticated client (the `groomer_delete` policy permits her own-row deletes).

### 2. Log Groom

**Trigger:** Add Appointment flipped, mini-verified, and Russell approves.

**Enable:** set `TIDYTAILS_ENABLE_LOG_GROOM_WRITE=on` (Production) + `vercel --prod`.

**Verify (mini-verifier):**
- Log one test groom ‚Üí exactly **one** new `appointments` row with
  `status='completed'` and `groomer_id` = Samantha's UID.
- Anon write still rejected.
- Reminder and Add Household **still gated**.

**Disable:** `vercel env rm TIDYTAILS_ENABLE_LOG_GROOM_WRITE production` +
`vercel --prod`.

### 3. Reminder send

**Trigger:** Log Groom flipped, mini-verified, and Russell approves ‚Äî **and**
the confirmed-send action is built (today `reminders.ts` only *prepares* a
draft; there is no send action to un-gate). Building it adds a
`confirmReminderSend` server action gated by `isReminderSendEnabled()`, plus the
Twilio wiring below.

**Twilio requirements (must all be true before this flip):**
- The `send-sms` Supabase edge function is deployed and its Twilio secrets live
  in Supabase function secrets ‚Äî **not** in the repo, never committed.
- The confirmed-send path calls `supabase.functions.invoke("send-sms", ‚Ä¶)` and
  writes one `automations_log` row ‚Äî **only** in response to an explicit,
  per-message operator confirmation.
- **HARD product rule (REQ-31):** Samantha reviews and explicitly confirms every
  single SMS. No automatic send, no batch send, no scheduled/background send ‚Äî
  ever. There is no cron, no queue, no "send all".

**Verify (mini-verifier):** a test SMS goes out **only** on an explicit
per-message confirm; no message sends without it; exactly **one**
`automations_log` row appears, scoped to Samantha; no scheduler/batch code path
exists anywhere; Add Household **still gated**.

**Disable:** `vercel env rm TIDYTAILS_ENABLE_REMINDER_SEND production` +
`vercel --prod`. **Tripwire:** any SMS Samantha did not confirm is a REQ-31
violation ‚Äî disable immediately and treat it as a watch-week rollback trigger.

### 4. Add Household

**Trigger:** Reminder flipped, mini-verified, approved ‚Äî **and** the
transactional RPC `create_client_with_first_pet` is rehearsed and applied. See
`2026-05-18-add-household-rpc-plan.md` and `2026-05-18-add-household-rpc.sql`.
Add Household must not be flipped on the two-insert path ‚Äî that orphans a client
row on partial failure (REQ-30).

**Enable:** apply the RPC migration (its own approval), then set
`TIDYTAILS_ENABLE_ADD_HOUSEHOLD_WRITE=on` (Production) + `vercel --prod`.

**Verify (mini-verifier):** create one test household ‚Üí exactly **one**
`clients` row + **one** `pets` row, both `groomer_id` = Samantha's UID,
`pets.client_id` correct ‚Üí force the pet insert to fail and confirm **no orphan
client** remains ‚Üí anon write rejected.

**Disable:** `vercel env rm TIDYTAILS_ENABLE_ADD_HOUSEHOLD_WRITE production` +
`vercel --prod`.
