# KoyaOS dogfood lesson — Tidy Tails v2 M3.1 live-data wiring

Earlier sessions produced REQ-01 through REQ-26. Wiring the v2 fast-lookup
wedge to read live Supabase data surfaced one more — a direct refinement of
REQ-12 (the fixtures↔live dispatch pattern).

---

## REQ-27 — Schema-contract guard for any deferred live-data path

**What happened:** The Ship 2.1 scaffold built the REQ-12 pattern — anonymized
fixtures as the committed default, a live `SELECT`-only path behind an env
flag. The live path was *deferred*: it was written but never the exercised
default, so nothing ran it. Its row mappers in `repo.ts` were written against
the **design-lock spec's assumed schema** (`appointments.service`,
`appointments.price`, `pets.typical_fee`). The **actual** live v1 schema uses
`service_type`, `fee`, `standard_fee`.

Because the mapper combined `select("*")` with non-throwing field access
(`str(r.service)` returns `""` when `r.service` is `undefined`), the mismatch
**never threw**. Live mode "worked" — it returned rows, raised no error — but
every appointment came back with a blank service and a `$0` price, and the
reports page showed `$0` revenue. The gap sat undiscovered from Ship 2.1
(2026-05-15) to M3.1 (2026-05-17): the path that was supposed to be the
scaffold's bridge to real data was silently emitting wrong data the whole time,
and nothing could fail loud because nothing exercised it and the access pattern
swallowed the drift.

**Why this matters:** REQ-12 codified the fixtures↔live dispatch layer as a
recommended pattern. This session shows the pattern has a sharp edge: a
deferred live path that fails *silently* is worse than no abstraction — it
manufactures false confidence ("the live path exists, it's wired") while
producing data that is quietly, plausibly wrong (`$0`, not a crash).

**What Koya needs:** When a scaffold includes a **deferred** live-data path —
built but not yet the exercised default — that path must carry a
**schema-contract guarantee that fails loud on column drift**. Two acceptable
forms:

1. **Generated types** — derive the domain types / row mappers from the live
   schema (e.g. `supabase gen types`), so a column rename becomes a compile
   error, not a `$0`.
2. **A schema-contract test** — assert the expected columns exist on the live
   table; fail CI when they don't.

Koya should also recommend the structural fix that made the contract testable
here: extract row mappers into a **pure module** (synthetic-row unit tests, no
DB) separate from the Supabase I/O glue. That covers mapper *logic* — but note
it does **not** cover schema drift on its own; the contract check above must
touch the real schema (generated types or a live-table assertion). When Koya
scaffolds the REQ-12 dispatch layer, it should scaffold the contract check at
the fixtures↔live seam in the same step.
