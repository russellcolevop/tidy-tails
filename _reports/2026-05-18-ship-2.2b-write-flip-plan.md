# Ship 2.2b — post-cutover write-surface flip plan

After the RLS cutover, v2 reads Samantha's live data but **every write surface
stays gated**. This plan sequences turning the four write surfaces on — one at a
time, across the watch week. It elaborates checklist §G with the actual code.

> **This plan does not authorize any flip.** Each flip is a separate code change
> needing its own explicit approval and its own mini-verifier. Nothing is
> flipped by writing or approving this document.

> separate, individually-approved act — nothing is enabled.

## Why writes are separate from the RLS cutover

The Ship 2.2b migration makes writes **safe** (RLS scopes them to Samantha) — it
does not make them **active**. The two are deliberately decoupled: a database
that is secure is not the same as an app that is mutating production. The
cutover and the flips are different decisions on different days.

## The write gate today — and the recommended flip mechanism

Each action (`createBooking`, `logGroom`, `prepareReminder`, `saveIntake`) runs
its full flow — auth re-check, validation, ownership check, payload build — then
in live mode returns a hardcoded `gated` result instead of persisting. There is
**no per-surface runtime flag today**; the gate is the `gated` branch in code.

**Recommended mechanism for each flip:** introduce a per-surface environment
flag, e.g. `TIDYTAILS_ENABLE_ADD_APPOINTMENT_WRITE`, default unset/off. The flip code
change adds the real `.insert()` path *behind that flag* — flag off → the action
still returns `gated`; flag on → it persists. This makes:

- **Enable** = set the flag in the Vercel project env + redeploy.
- **Disable** = unset the flag + redeploy — no code revert, seconds not minutes.

This refines checklist §G ("one code change") into "one code change that is
itself flag-gated" — strictly safer for the disable path the watch week needs.
The simpler fallback (no flag; the flip *is* the code change, disable = revert
the commit + redeploy) remains valid if Russell prefers fewer flags.

## `groomer_id` on inserts — rely on the DEFAULT

The migration gives every in-scope table `groomer_id uuid DEFAULT auth.uid()`.
v2's server Supabase client carries Samantha's JWT, so on an authenticated
insert `auth.uid()` is her UID and `groomer_id` is auto-stamped — satisfying the
`groomer_insert` policy `WITH CHECK (groomer_id = auth.uid())`. **The branch
rehearsal proved this**: the data-preservation probe row was inserted by the
authenticated user with no explicit `groomer_id` and was correctly stamped.

So the `build*Insert` payload builders do **not** need a `groomer_id` field.
Setting it explicitly is optional belt-and-suspenders; relying on the DEFAULT is
the proven, smaller change. Either way, never set `groomer_id` to a hardcoded
literal — it must be `auth.uid()` so the policy passes.

## Sequencing principles

- **One surface at a time**, spaced across the ~7-day watch week. Never a
  big-bang.
- **Order = operator-confidence order** (smallest/safest first), not technical
  difficulty: Add Appointment → Log Groom → Reminder send → Add Household.
- Each flip = **one code change + one `docs/DECISIONS.md` entry + one
  mini-verifier**.
- **No cascade (REQ-29):** every mini-verifier must re-confirm that the
  not-yet-flipped surfaces are *still gated*. A flip that accidentally enables
  another surface is a failed flip — disable and reassess.
- A flip happens only after the previous surface is flipped **and** its
  mini-verifier passed **and** Russell approves the next one.

## Surface 1 — Add Appointment  (first)

Smallest, best-rehearsed write (M2). Trigger: cutover is green (runbook §3).

**Code files:**
- `v2/lib/actions/appointments.ts` — `createBooking()`; replace the live `gated`
  branch (lines ~116–122) with the flag-gated `.insert()` into `appointments`.
- `v2/lib/actions/appointments.ts` — `BookingState` type: add a `saved` variant
  distinct from the dry-run `demo`.
- `v2/components/AddAppointment.tsx` — render the new `saved` state (success
  confirmation distinct from the demo dry-run echo).
- `v2/lib/booking.ts` — `buildAppointmentInsert`: no change needed (`groomer_id`
  via DEFAULT); touch only if setting `groomer_id` explicitly.
- `v2/.env.local.example` + Vercel env — add `TIDYTAILS_ENABLE_ADD_APPOINTMENT_WRITE`.

**The change:** in live mode + flag on, call
`const supabase = await createServerSupabase(); const { error } = await supabase.from("appointments").insert(payload);`
then `revalidatePath(\`/clients/${booking.client_id}\`)`; return `saved`. On
`error`, return a `formError` — nothing partially written (single-row insert).

**Mini-verifier:** book one real test appointment → exactly **one** new
`appointments` row → its `groomer_id` equals Samantha's UID
(`88413167-0799-49a7-ba4c-c1c29403e038`) → live count moves `baseline → +1` → an
anon REST INSERT to `appointments` is still rejected → Log Groom / Reminder /
Add Household **still return `gated`**.

**Disable / rollback:** unset `TIDYTAILS_ENABLE_ADD_APPOINTMENT_WRITE` + redeploy (action
returns to `gated`). If a bad row was written, delete it as Samantha via the
authenticated client (the `groomer_delete` policy now permits her own-row
deletes) — there is no bulk exposure. This surface does not, on its own, warrant
an RLS rollback.

## Surface 3 — Reminder send  (third) — human-confirmed, never automatic

Trigger: Surface 2 flipped and mini-verified; Russell approves.

> "Third" is operator-confidence order, not "third thing wired to Twilio." There
> is **no automatic, batch, scheduled, or background send — ever** (HARD product
> rule, REQ-31). The app *prepares* a reminder; Samantha reviews and explicitly
> confirms each individual SMS in-app.

**Important:** today `reminders.ts` only *prepares* a draft — there is no send
action at all. This flip **adds** the confirmed-send capability; it does not
just un-gate an existing one.

**Code files:**
- `v2/lib/actions/reminders.ts` — keep `prepareReminder()` as-is (it stays a
  pure, inert prepare). **Add a new server action** — e.g. `confirmReminderSend`
  — that takes the reviewed draft and, in live mode + flag on, invokes the
  `send-sms` edge function and writes one `automations_log` row. It is callable
  **only** from an explicit per-message confirm control.
- `v2/components/ClientActions.tsx` (and/or the reminder review UI) — wire a
  confirm control that calls `confirmReminderSend` only on an explicit tap;
  render a sent-confirmation state.
- `v2/.env.local.example` + Vercel env — add `TIDYTAILS_ENABLE_REMINDER_SEND`.
- Confirm the `send-sms` edge function's request contract before wiring (it is
  Twilio-backed, set up 2026-04; its secrets live in Supabase function secrets,
  not the repo).

**The change (confirmed-send path only):**
`await supabase.functions.invoke("send-sms", { body: { to: draft.to, message: draft.message } });`
then `await supabase.from("automations_log").insert({ client_id, type: "reminder", channel: "sms", message: draft.message });`
— executed **only** in response to the per-message confirm.

**Mini-verifier:** a test SMS goes out **only** on an explicit per-message
confirm → no message sends without it → exactly **one** `automations_log` row
appears, scoped to Samantha → no scheduler/batch/cron code path exists anywhere
→ Add Household **still gated**.

**Disable / rollback:** unset `TIDYTAILS_ENABLE_REMINDER_SEND` + redeploy. **Hard
tripwire:** if any SMS goes out that Samantha did not confirm, that is a REQ-31
violation — disable the surface immediately and treat it as a watch-week
rollback trigger (runbook §4.1).

## Surface 4 — Add Household  (last) — blocked on a transactional wrapper

Trigger: Surface 3 flipped and mini-verified; Russell approves; **and** the
transactional wrapper below exists and is itself verified.

**Precondition (REQ-30) — blocking:** the live path is a `clients` insert plus a
dependent `pets` insert (`pets.client_id` = the new client's id). The two are
**not atomic** in PostgREST — a pet-insert failure after a client-insert success
**orphans a client row**. This flip is blocked until one of:

- **(Recommended) a Postgres RPC** — a `SECURITY INVOKER` function that inserts
  client + pet in one transaction. This is itself a **schema change** needing
  its own migration, its own branch rehearsal, and its own approval — it is not
  part of the Ship 2.2b cutover. Track it as a separate small ship.
- **or a client-side revert** — on pet-insert failure, delete the just-created
  client row (now possible: the `groomer_delete` policy permits Samantha's
  own-row deletes). Simpler, but a crash between the two calls still orphans.

**Code files:**
- A new migration for the RPC (separate ship) — if the RPC path is chosen.
- `v2/lib/actions/intake.ts` — `saveIntake()`; replace the live `gated` branch
  (lines ~114–121) with a call to the RPC (or the revert-on-failure sequence).
- `v2/lib/actions/intake.ts` — `IntakeState` type: add a `saved` variant.
- `v2/components/AddHousehold.tsx` — render the `saved` state.
- `v2/lib/intake.ts` — `buildClientInsert` / `buildPetInsert`: no `groomer_id`
  change needed (DEFAULT handles both rows).
- `v2/.env.local.example` + Vercel env — add `TIDYTAILS_ENABLE_ADD_HOUSEHOLD_WRITE`.

**Mini-verifier:** create one test household → exactly **one** `clients` row +
**one** `pets` row, both `groomer_id` = Samantha's UID, `pets.client_id`
correct → **force the pet insert to fail and confirm no orphan client remains**
→ anon write rejected.

**Disable / rollback:** unset `TIDYTAILS_ENABLE_ADD_HOUSEHOLD_WRITE` + redeploy. Note
`clients.address` is captured by intake but not yet surfaced in the UI — that is
a post-cutover display follow-up, not a flip blocker (see the doc-gaps note).
