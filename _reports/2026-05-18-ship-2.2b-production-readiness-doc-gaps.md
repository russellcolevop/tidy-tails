# Ship 2.2b — production-readiness doc-gap resolutions

Five known doc/readiness gaps were reviewed for the production cutover. Each is
resolved below: a decision with rationale, and whether it blocks the cutover.

**Summary: none of the five blocks the Ship 2.2b cutover.** Two are display
follow-ups, two are scope confirmations, one is a documentation-hygiene
resolution.

---

## 1. Stale hardcoded row counts — RESOLVED (manifest is the only baseline)

**Gap:** older docs carry pre-reconciliation counts (`268` clients / `352` pets
/ `730` appointments) that no longer match reality (`137` / `188` / `737`).

**Resolution:** the principle is already settled and enforced — *the one and
only cutover baseline is the exact `count(*)` captured into the fresh backup
`MANIFEST.json` at cutover time*; never a number in prose, never a
`list_tables` planner estimate (which returned `appointments ≈ 7` against a true
`737` in this project — see the checklist Appendix).

Active cutover docs verified compliant:

- `2026-05-18-ship-2.2b-production-cutover-runbook.md` — every count gate
  compares against `MANIFEST.json`; states "never against a number in any
  planning doc."
- `2026-05-18-ship-2.2b-write-flip-plan.md` — uses "baseline → +1", no literals.
- `2026-05-17-ship-2.2b-backup-rollback-rehearsal-checklist.md` — its Appendix
  explicitly lists every stale source (including the Ship 2.2 plan §7 and the
  production-mode plan §4.5) and rules them out as baselines.
- The forward/rollback SQL and the UID record contain no row counts.

The hardcoded `268/352/730` figures survive only in **pre-reconciliation
historical docs** (`2026-05-13-open-questions.md`,
`2026-05-14-data-reconciliation-recovery-note.md`). Those are not cutover docs
and were accurate when written; they are correctly superseded and are **left
unedited** (rewriting historical records is not appropriate). No active cutover
doc presents a hardcoded count as a verifier baseline. **No action needed; no
blocker.**

## 2. Payments / Stripe — RESOLVED (explicitly out of scope, no contradiction)

**Resolution:** the v2.0 design-lock spec already states "No Stripe at v2.0 …
post-LICENSEABLE_READY unless Russell explicitly opens a design decision."
Confirmed for the cutover record:

- v2.0 **tracks money** — appointment `fee` / `tip` / `net` are recorded and

- No active cutover doc contradicts this. **No action needed; no blocker.**

## 3. Excel / CSV export-import — RESOLVED (later operator feature)

**Resolution:**

- **CSV export** (appointment-list export, monthly revenue export, a danger-zone
  "export all data") is designed in the design-lock spec but **not yet built**.
  It is a **post-cutover operator-convenience feature** — it reads data, never
  writes, so it carries no cutover risk and is **not cutover-blocking**. Build
  it after the watch week.
- **CSV/Excel import** is **not in v2.0 scope**. The only historical use of
  Excel was the one-off contact-card reconciliation (an intermediate
  spreadsheet, not an app feature). An import loop, if ever wanted, is a
  separate future ship. **No action needed; no blocker.**

## 4. `clients.address` captured but not surfaced — DECISION: post-cutover follow-up

**Finding:** the v2 intake form captures `address` (`buildClientInsert`
includes it; the `clients.address` column exists), but no v2 screen renders it —
the client detail page shows name, phone, alt-contact, and notes, not address.

**Decision: post-cutover follow-up — NOT cutover-blocking.**

Rationale: this is a pure **display gap**. The data is captured and stored
correctly; it is simply not shown. It has no security, data-integrity, RLS, or
workflow-blocking dimension, and the RLS cutover does not touch it. Surfacing
`address` on the client detail page is a small read-only UI addition for after
the watch week. Tracked here as a post-cutover follow-up.

## 5. `lastAppointment` future-date behaviour — DECISION: fix post-cutover, before the Add Appointment flip

**Finding:** `v2/lib/derive.ts` `lastAppointment()` returns the
maximum-`date` appointment with **no past/future filter**. Consequences:

- "Last visit" (home page and client detail) would display a **future** date
  for any client who has a future-dated appointment.
- `daysSinceLastVisit()` returns a **negative** number for such a client; in
  `lapsedClients()` that client is then excluded from the lapsed list
  (conservative — they have an upcoming visit — but the displayed "last visit"
  is still wrong).

**Risk:** this is a **display-correctness** bug, not data corruption — no row is
wrong, the derivation is. It is currently **latent**: v1's data is overwhelmingly
past-dated grooms, and v2 writes are gated. It becomes *active* only once
future-dated `appointments` rows exist — which is exactly what the **Add
Appointment** write flip introduces (a booking is a future-dated appointment).

**Decision: NOT cutover-blocking; fix it post-cutover but BEFORE the Add
Appointment write flip (write-flip plan Surface 1).**

Rationale: the RLS cutover is read-behaviour-preserving and does not trigger this
bug. The bug only manifests when future-dated rows appear, and the Add
Appointment flip is the first thing that creates them — so the fix is a
precondition of Surface 1, not of the cutover. The fix is small: "last visit"
should mean the most recent **past/today** appointment (or most recent
`status='completed'`), with future bookings surfaced separately as a "next
appointment". Tracked here; to be scheduled into the Surface 1 flip work.
