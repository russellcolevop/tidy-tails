# KoyaOS dogfood lesson — Tidy Tails v2 Log Groom (second write surface)

Earlier sessions produced REQ-01 through REQ-28. Building Log Groom — the
venture's second write surface, and the point where the production-mode plan
made cutover sequencing concrete — surfaced one more.

---

## REQ-29 — Per-surface write flags must be independent of the global security gate

**What happened:** Log Groom is Tidy Tails v2's second write surface, built —
like M2 (REQ-28) — in dry-run/gated mode: the complete flow runs (auth,
validation, ownership, payload), nothing persists. Both M2 (Add Appointment) and
Log Groom now sit behind the *same* global security gate: the Ship 2.2b RLS
cutover. The production-mode implementation plan written this session made the
consequence explicit — when that gate lifts, the surfaces are switched to live
**one at a time**, each its own logged decision, not all at once.

**Pattern that worked:** two gate layers, modeled and operated as distinct
things:

1. the **global store-safety gate** — "is the database safe to write to" — the
   RLS-hardening migration. Atomic, one event, venture-wide.
2. a **per-surface write flag** — "is *this* surface's write active" — one per
   write surface (Add Appointment, Log Groom, …), each lifted by its own
   explicit, logged decision.

The cutover makes writes *safe*; it does not make them *active*. Each surface
stays independently flippable, with its own first-real-write verifier.

**Why it matters:** REQ-28 codified the dry-run pattern for a venture's *first*
write surface, and said "lifting the gate is one explicit, logged decision."
With one surface, "the gate" and "the surface" are the same thing — the
distinction is invisible. The second surface makes it visible: a venture in
production-mode build accumulates **N dry-run write surfaces behind one security
gate**. If the platform models that as a single flag, lifting the security gate
cascades every surface live at once — the big-bang flip REQ-28 exists to
prevent. "One logged decision" has to mean one *per surface*, not one *for all*.

**What Koya needs:** model the two layers separately and never cascade one into
the other.

1. Track a **per-surface write flag** distinct from the global store-safety gate.
2. When Koya detects the security gate lifting (e.g. an RLS-hardening migration
   landing), it must **not** auto-activate the write surfaces — each stays its
   own explicit `docs/DECISIONS.md` flip.
3. Support a **surface-by-surface activation sequence**: the first real write on
   each surface is its own small, audited event with its own verifier (one row
   appears, scoped to the operator; an anonymous write is still rejected).

**Relation to prior REQs:** REQ-12 is the fixtures/live *read* dispatch; REQ-27
guards a deferred read path against schema drift; REQ-28 is the dry-run pattern
for the *first* write surface. REQ-29 is the multi-surface generalization — once
a venture has several dry-run surfaces behind one cutover, their activation must
be per-surface and decoupled from the single security event.
