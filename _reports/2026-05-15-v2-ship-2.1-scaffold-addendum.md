## 3. Safety gates (hard — non-negotiable for this ship)

- **No live Supabase mutation.** The live data path is SELECT-only and OFF by default.
- **No SQL execution** against the live project.
- **No SMS sends.** The reminder UI is built, but its Send action is disabled behind a gate notice.
- **No schema or RLS changes.**
- **No v1 changes.** v1 HTML modules, the repo root, and GitHub Pages config are untouched.
- **No customer data committed.** Fixtures are fully synthetic. `.env.local` is gitignored.
- **No deploy.** `localhost:3000` only until Russell explicitly approves.

## 4. Scope — what Ship 2.1 builds

**Fully functional, read-only (data from fixtures, or live SELECT when the flag is on):**

- Search-first home — fast client/pet search by name or phone
- Client detail — pet safety cards (allergy-first), contact, appointment history
- Pet detail — allergy + grooming notes, vaccination status, fee memory
- Appointment history per client and per pet
- Lapsed clients view
- Vaccination status (expiring / expired surfacing)
- Revenue visibility — monthly totals, appointment list
- Tiny settings — read-only display of profile / SMS template / lapsed threshold
- PWA baseline — manifest, app icon placeholders, service worker, installability, mobile viewport
- Placeholder auth gate — demonstrates the persistent-session UX

**Built as UI but write-disabled (gated, visible so Samantha can see the full shape):**

- Quick groom logging modal — present; Save disabled with a "writes gated until auth + RLS" notice
- SMS reminder sheet — present; Send disabled with the same gate notice

**Explicitly NOT in this ship:** real authentication, any write path, SMS delivery, schema/RLS
migration, Vercel deploy, full offline write queue, and any Pawfinity-style admin module
(calendar, booking, POS, inventory, payroll, expenses).

## 5. App structure (`v2/`)

```
v2/
  app/
    layout.tsx                      root layout, metadata, manifest link, viewport
    globals.css                     Tailwind + design tokens
    page.tsx                        entry — routes to login or app
    (auth)/login/page.tsx           placeholder login gate
    (app)/layout.tsx                authenticated shell + bottom nav
    (app)/page.tsx                  search-first home
    (app)/clients/[id]/page.tsx     client detail
    (app)/clients/[id]/pets/[petId]/page.tsx   pet detail
    (app)/reports/page.tsx          revenue + lapsed clients + vaccinations
    (app)/settings/page.tsx         tiny read-only settings
  components/                       ClientSearch, PetCard, AllergyAlert, AppointmentHistory,
                                    VaccinationList, LogGroomSheet, SendReminderSheet,
                                    GatedActionNotice, BottomNav, ...
  lib/
    data/types.ts                   Client / Pet / Appointment / Vaccination types
    data/fixtures.ts                anonymized synthetic seed data
    data/repo.ts                    data access — fixtures OR live SELECT (flag-gated)
    data/supabase.ts                read-only Supabase client (dormant unless flag on)
    format.ts                       date / money formatting
    derive.ts                       lapsed-client, vaccination-status, revenue computations
  middleware.ts                     placeholder auth-cookie redirect
  public/                           manifest.webmanifest, icons/, sw.js
  README.md                         run / installability / deploy / safety notes
  .env.local.example                env shape (no secrets)
```

The data-model types mirror design-lock spec §6 so that swapping fixtures for live data is a
single-layer (`repo.ts`) change with no UI rework.

## 7. Run / verify

```
cd v2
npm install      # first time only
npm run dev      # → http://localhost:3000
```

Read-only by construction. To preview against live data locally (still SELECT-only):
copy `.env.local.example` to `.env.local`, set `NEXT_PUBLIC_USE_LIVE_DATA=on` plus the Supabase
URL and anon key, and restart the dev server. `.env.local` is gitignored.
