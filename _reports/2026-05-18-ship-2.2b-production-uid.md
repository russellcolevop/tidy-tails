# Ship 2.2b — Samantha's production auth UID

This file records the **production** `auth.users` identity used by the Ship 2.2b
RLS-hardening migration. It is a cutover input, captured so the production
forward migration backfills `groomer_id` to the correct owner.

A UUID is **not a secret** — it appears in JWTs and RLS policy expressions and
is safe to commit. No password, key, or token is recorded here.

## How it was captured

A single **read-only `SELECT`** against production `pgkwovokciaqnbhpttba`:

```sql
SELECT id, email, role, aud,
       (encrypted_password IS NOT NULL) AS has_password,
       email_confirmed_at IS NOT NULL AS email_confirmed,
       created_at, last_sign_in_at
FROM auth.users
WHERE email = 'sammclennan143@gmail.com';
```

No DDL, no write, no `apply_migration` — production schema and data are
unchanged. This is the same read-only-introspection category as the
2026-05-17 production preflight snapshot.

## Where this UID is used — exactly

1. **Production forward migration** (`2026-05-18-ship-2.2b-production-forward-migration.sql`),
   Step A backfill — one statement per in-scope table:
   `UPDATE public.<table> SET groomer_id = '88413167-0799-49a7-ba4c-c1c29403e038' WHERE groomer_id IS NULL;`
   for `clients`, `pets`, `appointments`, `booking_requests`, `client_accounts`,
   `automations_log`. After the migration every existing row's `groomer_id`
   equals this UID.

2. **At runtime** — when Samantha is signed into v2, `auth.uid()` returns this
   UID, so the `groomer_id = auth.uid()` RLS policies match her rows. The UID is
   **not** hard-coded into any policy or the column `DEFAULT` — the policies and
   the `DEFAULT auth.uid()` clause are dynamic.

3. **NOT used** in the rollback migration — rollback drops the `groomer_id`
   column entirely, so no UID appears there.
