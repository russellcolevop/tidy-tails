# Phase 3 — rollback-only test outcome (2026-05-16)

## What was run

The full Phase 3 transaction (Section 1 through Section 7 of `_reports/2026-05-16-phase-3-inserts.sql`) was submitted to Supabase MCP `execute_sql` with `ROLLBACK;` active. The script's intent was: 7 client INSERTs, 8 pet INSERTs, 0 appointment touches, then ROLLBACK.

## What happened

The transaction aborted partway through Section 2 (pre-write gates):

```
ERROR:  P0001: G5 FAIL: 1 client row(s) already exist with names matching the 7 proposed inserts.
CONTEXT:  PL/pgSQL function inline_code_block line 13 at RAISE
```

PostgreSQL automatically rolled back the transaction on the exception. Zero INSERTs executed. Zero rows changed.

Independent verification of live state post-test:

State matched the pre-test baseline exactly. Rollback was complete.

## Root cause — Korrie Silver already exists

A targeted query identified the G5 match:

```json
[{
  "id": "56f4385b-b103-4aab-86a3-f18219d7aabe",
  "first_name": "Korrie",
  "last_name": "Silver",
  "phone": "647-300-7952",
  "matched_clause": "korrie+silver"
}]
```

The same row carries:

- 8 appointments.
- Two pet rows both named "Gavi" — one breed `Medium Mix`, one `Yorkie Mix`, both with `allergies = false` and `allergies_detail = NULL`.

Why G2 (phone-collision gate) didn't catch it: the proposed Phase 3 phone was `1-647-300-7952` (Sam's spelling, with the `1-` country-code prefix). The existing row stores it as `647-300-7952`. String equality treats those as different. G5's paired-name check is the cross-check that surfaced the same-person collision.

## Verifier verdict

G5 did exactly its job. This is a planning-discipline win, not a script defect. The Codex verifier asked Cowork to:

1. Treat G5 as a successful safety gate.
2. Revise Phase 3 to hold out Korrie Silver / Gavi until row-level reconciliation lands separately.

## KoyaOS dogfood lesson

- **Phone-format collisions are a real failure mode of phone-only G2 gates.** A name-paired G5 gate is necessary defense-in-depth, not redundant ceremony. KoyaOS reconciliation-phase template: any phase that INSERTs new client rows should ship with both phone-collision and name-collision pre-write gates.
- **Hold-out patterns deserve their own gate.** Splitting G5 into G5a (cover the active set) + G5b (assert the held-out invariant is still as-expected) makes the script document the holdout rather than just route around it. If the held-out row drifts post-test, the script fails loud and the operator re-investigates.
