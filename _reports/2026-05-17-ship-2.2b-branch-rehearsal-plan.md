# Ship 2.2b — forward + rollback migration rehearsal plan

The Ship 2.2b RLS migration is one-way and reaches Samantha's live business the
moment it commits. It must be **rehearsed end-to-end before it is ever run on
production**. This plan turns checklist §D/§E into an executable sequence.

The verified service_role backup
`venture-ops/backups/tidy-tails/2026-05-18-rehearsal/` (1,131 rows across 7
tables — `clients` 137, `pets` 188, `appointments` 737, `booking_requests` /
`client_accounts` / `automations_log` 0, `sam_review_responses` 69; readback
PASS) is the data source and the count baseline for everything below.

---

## §0 — What this plan does NOT authorize

This is a **plan**. Approving it does **not** authorize: creating a Supabase
branch, creating a throwaway project, importing any data, running any SQL,
applying any migration, mutating production in any way, lifting a write gate, or
sending an SMS. Each of those is a separate, explicitly-approved step. The
production RLS migration is **out of scope entirely** — this plan stops at the
rehearsal.

---

## §1 — Rehearsal target: options, tradeoffs, recommendation

The migration's whole point is to close **R-1**: after it runs, an anonymous
REST request must return `[]` (HTTP 200, empty). That behavior is produced by
the *full stack* — PostgREST + JWT role resolution (`anon` vs `authenticated`) +
RLS policy evaluation + `auth.uid()`. **A rehearsal is only evidence if the
target reproduces that whole stack.** Fidelity is the gate; cost is only the
tiebreaker.

| Target | Reproduces Auth + PostgREST + RLS + `auth.uid()`? | Cost | Setup burden |
|---|---|---|---|
| **Supabase branch** | Yes — it *is* a Supabase project | Requires Pro-plan upgrade (this project is free tier) | Lowest — clones schema, policies, view, **and data** automatically |
| **Throwaway free Supabase project** | Yes — full stack, identical free/Nano tier to prod | None | Higher — schema, policies, view, and data must be recreated by hand |
| **Local Postgres** | **No** — has RLS, but no Supabase Auth, no PostgREST gateway, no real `anon`/`authenticated` JWT roles | None | Medium |

**Local Postgres is disqualified** as the rehearsal target. You can fake
`auth.uid()` with a stub function and test that the *policy SQL* parses and
evaluates — but you cannot test the thing that actually closes R-1: an anon REST
call returning `[]` through PostgREST. A green local rehearsal would be a false
signal. Local Postgres is fine only as a throwaway *syntax check* of the
migration SQL — never as the gate.

**Recommendation: a throwaway free-tier Supabase project.** It is the same Nano
free tier as production, so behavior is identical; it has the complete Auth +
PostgREST + RLS stack, so every R-1 check is faithful; and it costs nothing.

Its one real downside is setup burden: a fresh project is empty, so production's
pre-migration state — schema DDL, the permissive policies, the `client_overview`
view — and the 1,131 data rows must all be **recreated by hand** from the §C
snapshot and the backup, *before* the migration rehearsal can prove anything
(see §2.2/§2.3, and the dedicated verifier gate in §5).

**A Supabase branch is the lower-risk paid alternative.** A branch clones
schema, policies, view, and data automatically — eliminating the two
highest-risk setup steps (state recreation and data import). If Russell is
willing to put the project on the Pro plan for the rehearsal window, a branch is
materially less error-prone. Exact branch/project cost can be confirmed with the
Supabase `get_cost` tool if wanted — this plan does not assume a dollar figure.

> **Checklist correction:** checklist §D is written branch-first ("Create a
> Supabase branch from production"). Branching is not available on this
> project's current free tier. See §6 for the proposed §D amendment.

## §2 — Forward migration rehearsal

Sequence on the chosen target. Steps 2.1–2.4 are *environment setup*; the
migration itself is 2.5; 2.6 is verification.

### 2.1 — Capture the §C snapshot from production (read-only)

Per checklist §C, captured **read-only against production** — no mutation:

- Every RLS policy on all 7 public tables (`SELECT * FROM pg_policies WHERE
  schemaname='public'`): name, cmd, roles, `qual`, `with_check`.
- The `client_overview` view definition (`pg_get_viewdef('public.client_overview', true)`).
- Full table DDL / column definitions for all 6 in-scope tables + `sam_review_responses`.
- `get_advisors` (security) output — the *before* state.

Save as dated `_reports/` artifacts (policy/view/schema only — **no row data**).
This snapshot is dual-purpose: it is both the raw material for the rollback
(§3) **and** the spec for recreating the rehearsal target's starting state (2.2).

### 2.2 — Recreate production's pre-migration state on the target

*(Throwaway-project path. A branch clones this automatically — skip to 2.4.)*

Recreate, exactly as captured in 2.1: the table DDL for all 7 tables; RLS
enabled on each; every permissive policy verbatim (`qual=true`, `roles={public}`
— note the three DELETE policies were dropped 2026-04-22, so they are absent);
the `client_overview` view.

⚠️ **This is the highest-risk setup step.** If the recreation drifts — a missing
policy, a different `qual`, a wrong view definition — the migration rehearses
against a target that is *not* production's starting state, and "rehearsed
green" becomes a false signal. §5 makes "target matches the §C snapshot" a
discrete verifier gate that must pass **before** 2.5.

### 2.3 — Import the backup data

Load the 1,131 rows from `venture-ops/backups/tidy-tails/2026-05-18-rehearsal/`
into the target. The backup is **JSON** (per the `dump_supabase.py` contract),
so this needs a JSON→SQL loader — a small companion script, or a one-off
conversion. *(Authoring that loader is a prerequisite sub-task, tracked
separately; it is not part of this planning step.)* A Supabase branch sidesteps
this entirely by cloning data.

The checklist §D warning applies: **confirm the target carries data before
rehearsing.** Against an empty table the backfill and the count verifier pass
*vacuously* and prove nothing.

### 2.4 — Create the rehearsal test user

Create one Supabase Auth user on the target (the rehearsal's stand-in for
Samantha) and capture its `auth.users` UID — call it `:rehearsal_uid`.

⚠️ **`:rehearsal_uid` is NOT Samantha's production UID.** A fresh project has
its own `auth.users`; production's UID does not exist there. The rehearsal
substitutes `:rehearsal_uid` into the migration. The *real* cutover substitutes
Samantha's verified **production** UID. These must never be confused — see §4
and the hard gate in §5's preflight.

### 2.5 — Apply the forward migration

Apply the Ship 2.2b forward migration — **parent plan §4, Steps A → B → C, as
one named Supabase migration** — to the target, with `:samantha_uid` replaced by
`:rehearsal_uid`. Do not restate the SQL here; §4 is authoritative. In short:

- **A** — `ADD COLUMN groomer_id uuid REFERENCES auth.users(id) DEFAULT
  auth.uid()` on the 6 in-scope tables; backfill every row to `:rehearsal_uid`;
  `SET NOT NULL`.
- **B** — drop every permissive policy (names from the 2.1 snapshot); create the
  four `groomer_*` `auth.uid()`-scoped policies per table.
- **C** — `DROP VIEW IF EXISTS public.client_overview`.

`sam_review_responses` is **left untouched** — it is out of the migration's
6-table scope (parent plan §4).

### 2.6 — Verify the forward migration (see §5 phases 2–4)

`groomer_id` present + backfilled + `NOT NULL` on all 6 tables; only the four
`groomer_*` policies remain, no permissive policy survives; `client_overview`
dropped; row counts unchanged vs the manifest baseline; v2 authenticated reads
work; anon reads/writes are blocked.

## §3 — Rollback migration rehearsal

Rehearsed on the **same target, immediately after** the forward rehearsal
(forward → then rollback). Authored from the §C snapshot, this is the inverse
migration (parent plan §5):

- **C⁻¹** — recreate `client_overview` from the captured view definition.
- **B⁻¹** — drop the four `groomer_*` policies on each table; recreate the
  captured permissive policies (`qual=true`, `roles={public}` for
  SELECT/INSERT/UPDATE — DELETE stays dropped, matching the 2026-04-22 state).
- **A⁻¹** — `DROP COLUMN groomer_id` on all 6 tables.

Save the rollback SQL as a dated artifact alongside the forward migration —
reviewed by Russell, ready to paste at cutover if the watch week needs it.

**Data-preservation step:** between the forward and rollback steps, create one
test row on the target. It must survive the rollback — the rollback drops a
*column*, never *rows*.

### "Rollback rehearsed green" — exact criteria

After forward → rollback on the target, **all** of the following hold:

1. Per-table row counts match the manifest baseline **exactly** (137 / 188 /
   737 / 0 / 0 / 0; `sam_review_responses` 69) — *plus* the one test row in
   whichever table it was added to.
2. `client_overview` exists again — **byte-compare** its definition against the
   §C capture.
3. `groomer_id` is absent from all 6 tables.
4. The original permissive policies are back (`qual=true`, `roles={public}` for
   SELECT/INSERT/UPDATE; DELETE absent).
5. An anon REST request returns **rows again** — the inverse of the R-1-closed
   test; proves v1 would work after a real rollback.
6. `get_advisors` (security) shows the original pre-migration finding set — the
   state is genuinely restored, not merely "no error."
7. The test row from the data-preservation step survived.

## §4 — Credentials & IDs needed

| Item | For | Where it comes from | Boundary |
|---|---|---|---|
| Samantha's **production** `auth.users` UID + email | The **real cutover** migration backfill (NOT the rehearsal) | Captured read-only from production at pre-flight (checklist §A.2) | A UID is not a secret; record it in the backup manifest. |
| `:rehearsal_uid` — the test user's UID | The **rehearsal** migration backfill | Created in step 2.4 on the target | Rehearsal-only. Never substituted into a production migration. |
| Target project URL + **anon** key | Pointing v2 at the target; the anon-rejection checks | The branch / throwaway project's API settings | Anon key is public-safe. |
| Target **service_role** key | Importing data (2.3) and applying migrations on the target | The target project's API settings | Never written to the repo, a tracked file, the manifest, or logs. Held in the operator's process env only — same rule the backup followed. |
| `v2/.env.local` values | Running v2 against the target | Set by the operator | `NEXT_PUBLIC_SUPABASE_URL` = target URL; `NEXT_PUBLIC_SUPABASE_ANON_KEY` = target anon key; `NEXT_PUBLIC_USE_LIVE_DATA=on`. `.env.local` is gitignored — never commit a filled-in copy. |

### service_role usage boundaries

- service_role is used **only** to import data and apply migrations **on the
  rehearsal target** — never against production in this plan.
- service_role **bypasses RLS**, so it can never be used for the anon-rejection
  checks (§5 phase 4). Those must use the target's **anon** key with no session
  — that is the only key that actually exercises the R-1 surface.
- The rehearsal target's service_role key is a *different* secret from
  production's. Neither is ever persisted to disk or the repo.

## §5 — Six-phase verifier checklist

Work top to bottom. A phase must be fully green before the next begins.

### Phase 1 — Preflight

- [ ] Verified backup present and readback-clean (`2026-05-18-rehearsal/`,
      done).
- [ ] §C snapshot captured from production read-only (policies, view, schema,
      advisors) and saved as dated `_reports/` artifacts.
- [ ] Rehearsal target chosen and created (throwaway project, or branch).
- [ ] **Target state matches the §C snapshot** — byte-compare every recreated
      policy `qual`/`with_check` and the `client_overview` definition against
      the 2.1 capture. *This gate stands between setup and the migration; if it
      is not green, nothing below proves anything.*
- [ ] Backup data imported; per-table counts on the target equal the manifest
      baseline (137 / 188 / 737 / 0 / 0 / 0 / 69).
- [ ] Test user created on the target; `:rehearsal_uid` captured.
- [ ] **Hard gate for the real cutover (recorded here, enforced at §F):** the
      UID substituted into the *production* migration is Samantha's verified
      production `auth.users` UID — **never** `:rehearsal_uid`.

### Phase 2 — Forward migration

- [ ] Migration applied as one named Supabase migration; it committed atomically.
- [ ] `groomer_id` exists on all 6 in-scope tables; zero NULLs; `NOT NULL`
      enforced; `DEFAULT auth.uid()` present.
- [ ] Every existing row's `groomer_id` equals `:rehearsal_uid`.
- [ ] Each of the 6 tables has exactly the four `groomer_*` policies; **no**
      policy with `qual=true` / `roles={public}` survives (`pg_policies`).
- [ ] `client_overview` view no longer exists.
- [ ] Per-table row counts unchanged vs the manifest baseline.
- [ ] `sam_review_responses` untouched — still 69 rows, original policy.

### Phase 3 — v2 authenticated read check

- [ ] `v2/.env.local` points at the target (URL + anon key, `USE_LIVE_DATA=on`).
- [ ] Log in to v2 as the test user; the session is established.
- [ ] Client list, client detail, pet detail, and reports all load with data.
- [ ] Counts/figures match the manifest baseline; revenue total spot-checked.
- [ ] *(Why this works: v2's live reads go through the cookie-bound
      authenticated server client, so `auth.uid()` = `:rehearsal_uid`, matching
      every row's backfilled `groomer_id` → `groomer_select` passes.)*

### Phase 4 — Anon rejection check (the R-1 surface)

- [ ] Anon-key REST `GET` on `clients` / `pets` / `appointments` returns `[]` —
      HTTP 200, empty (not 401/403). This is the R-1-closed definition.
- [ ] Anon-key `INSERT` / `UPDATE` / `DELETE` on any of the 6 tables is
      rejected.
- [ ] A logged-out v2 instance pointed at the target redirects to `/login`
      (proxy) and renders no data.

### Phase 5 — Rollback migration

- [ ] One test row created on the target (data-preservation probe).
- [ ] Rollback migration applied (§3: C⁻¹ → B⁻¹ → A⁻¹).
- [ ] All seven "rollback rehearsed green" criteria (§3) hold.

### Phase 6 — Post-rollback checks

- [ ] `get_advisors` (security) matches the pre-migration *before* state.
- [ ] Anon REST returns rows again — v1-style behavior restored **on the target
      only**.
- [ ] The forward migration SQL and the rollback migration SQL are both saved as
      dated `_reports/` artifacts for Russell's review.
- [ ] Rehearsal result recorded as a dated artifact: what ran, pass/fail per
      gate, target used.

**"Rehearsal complete" = Phases 1–6 all green.** Only then is the forward
migration cleared to be drafted against production (checklist §F) — and even
then, never first-run there.
